# 015 - 项目加固与工程化优化

## 需求背景

当前项目核心功能已可用，但在权限安全、接口入参校验、数据库一致性、查询效率、类型收敛和工程化方面仍有较明显短板。继续在现状上叠加新功能，会持续放大维护成本和线上风险。

本次需求的目标不是新增业务能力，而是对现有系统进行一轮结构化加固，先解决安全与正确性问题，再提升性能与一致性，最后补齐工程基础设施。

## 优化目标

1. 将 RBAC 从前端展示控制补齐到后端接口强制校验
2. 统一 API 入参校验、鉴权入口和错误返回风格
3. 修复典型 N+1 查询和权限加载低效问题
4. 收敛数据库、接口、前端之间的类型边界
5. 建立更稳定的数据库演进、测试和文档维护机制

## 功能列表

### P0 - 安全与正确性

1. 为所有写操作 API 增加服务端权限校验
2. 为关键读操作 API 增加最小权限校验
3. 为 Route Handler 接入统一的 Zod 入参校验
4. 移除生产环境默认 JWT Secret，改为显式配置
5. 统一鉴权和授权工具，减少重复实现
6. 梳理注册流程中“用户已创建但邮件发送失败”的状态处理
7. 审查并收紧公开接口和免鉴权路径

### P1 - 性能与一致性

1. 修复用户列表接口的 N+1 查询
2. 优化登录接口中的角色/权限加载逻辑
3. 优化订阅详情等多表查询接口
4. 统一数据库主键与前端类型的表示方式
5. 抽离系统配置读取与 JSON 解析工具
6. 梳理外键、索引、唯一约束需求

### P2 - 工程化

1. 建立 Drizzle migration 流程
2. 将 seed 脚本收敛为“建表后初始化数据”的单一职责
3. 建立最小测试体系，覆盖认证、权限、节点解析、订阅生成等核心模块
4. 增加 lint / format 基础设施
5. 抽公共 API 工具层，统一分页、校验、鉴权和错误处理
6. 清理 AGENTS 与 specs 的文档漂移
7. 补齐缺失的 execution.md，修复已有 spec 编号与记录不一致问题

## 验收标准

### P0 验收

1. 无权限用户无法通过直接调用接口完成增删改操作
2. 关键只读接口具备最小必要权限校验
3. 非法参数请求返回统一中文错误消息
4. 生产环境缺少 JWT Secret 时不允许以默认值继续运行
5. 注册流程对邮件发送失败的行为明确且可预期

### P1 验收

1. 用户列表、登录、订阅详情等核心接口查询次数明显下降
2. 类型定义不再频繁依赖 `String()` / `Number()` 互转
3. 系统配置读取逻辑统一，不再散落多处 `JSON.parse`
4. 数据库约束设计与代码实现保持一致

### P2 验收

1. 表结构变更通过 migration 管理，不再依赖 seed 脚本建表
2. 核心模块具备基础测试覆盖
3. 代码规范有可执行的工具链约束
4. AGENTS 和 specs 能真实反映当前项目状态

## 非目标

1. 不在本次优化中引入新的业务模块
2. 不重写现有 UI 风格与页面布局
3. 不在本次优化中替换 Next.js、SQLite 或认证方案

## 涉及数据与配置

### 鉴权与权限

- Cookie：`token`
- JWT 配置：`JWT_SECRET`
- 用户、角色、权限、角色权限关系

### 业务数据

- 用户：`sys_user`
- 角色：`sys_role`
- 权限：`sys_permission`
- 用户角色：`sys_user_role`
- 角色权限：`sys_role_permission`
- 订阅、节点及其关联表

### 系统配置

- `sys_config.config_key = 'mail'`
- `sys_config.config_key = 'site'`

## 执行顺序

1. P0 安全与正确性
2. P1 性能与一致性
3. P2 工程化
